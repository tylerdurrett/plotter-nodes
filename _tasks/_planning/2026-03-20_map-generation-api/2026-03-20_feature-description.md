# Feature: Map Generation API

**Date:** 2026-03-20
**Status:** Scoped

## Overview

Add a FastAPI-based HTTP server to Portrait Map Lab that exposes map generation as an on-demand API. This eliminates the manual CLI-run-then-copy-files loop when experimenting with different map configurations, and establishes the foundation for real-time integration with the companion plotter repo and any future consumers.

## End-User Capabilities

1. Start a local API server with a single command (`uv run python scripts/run_pipeline.py serve`) that exposes all map generation capabilities over HTTP.
2. Submit an image (via multipart upload or local path reference) and receive generated maps as downloadable binary assets, with a manifest in the exact `ExportManifest` format the plotter repo's `parseManifest()` already consumes.
3. Request exactly the maps needed — from the full pipeline down to a single map like `importance` or `flow_x` — with granular control over which pipelines run and which outputs are returned. Requestable maps are the 7 keys defined in `_MAP_DEFINITIONS`: `density_target`, `flow_x`, `flow_y`, `importance`, `coherence`, `complexity`, `flow_speed`.
4. Override any pipeline parameter per-request (remap curves, weights, blend modes, complexity metrics, etc.) without modifying config files or CLI invocations.
5. Optionally persist a generated bundle to disk (identical to the existing `--export` CLI output), so good results can be saved for the file-based workflow or committed alongside sketches.
6. Reference the API documentation to understand endpoints, parameter schemas, and response formats well enough to build client integrations (e.g., the planned plotter-side integration).

## Architecture / Scope

### In Scope

- FastAPI application with `uvicorn` server, launched as a `serve` subcommand on the existing CLI entry point
- Image submission endpoint accepting both multipart file upload and local filesystem path (local path is a convenience for localhost development; upload ensures the API works identically when hosted remotely)
- Granular map generation endpoint where the caller specifies which maps to produce and with what parameters
- Response format: JSON containing the full `ExportManifest` structure (matching `_manifest_to_dict()` output) plus URLs to fetch individual `.bin` map files — so the plotter's `MapBundle.load()` can point at a session's base URL and work without changes
- Disk-backed session cache with TTL-based cleanup
- Optional disk persistence via request parameter (copies the cache directory to a named output path)
- Minimal pipeline orchestration logic that resolves which pipelines to run based on the requested maps, calling individual pipeline functions directly
- OpenAPI/Swagger docs auto-generated by FastAPI
- Health check and map-key enumeration endpoints
- A concise integration guide documenting how a client (specifically the plotter repo) would consume the API
- `.cache/` added to `.gitignore`

### Out of Scope

- Plotter repo client-side integration (separate initiative — this feature provides the documentation to make that straightforward)
- Preset/recipe YAML system (future feature that will layer on top of the API)
- Composable map recipe engine (future feature)
- Authentication, rate limiting, or multi-user concerns (this is a local development tool for now)
- WebSocket streaming or push-based updates
- New exportable map types (e.g., `tonal_target`) — extending `_MAP_DEFINITIONS` is a separate decision

### Relationship to Existing Systems

The API wraps the same pipeline functions the CLI already uses (`run_feature_distance_pipeline`, `run_contour_pipeline`, `run_density_pipeline`, `run_flow_pipeline`, `run_complexity_pipeline`) and the same export serialization (`build_export_bundle`, `_manifest_to_dict`). The one piece of new logic is the pipeline dependency resolver that determines the minimal set of pipelines to run for a given set of requested maps — the existing `run_all_pipelines` always runs everything, so the API calls the individual pipeline functions directly and shares landmarks across them.

## Technical Details

### Server

- **Framework**: FastAPI (async-capable, auto-generates OpenAPI docs, widely adopted)
- **Runner**: Uvicorn, started via CLI subcommand
- **Default port**: 8100 (avoids conflict with Vite at 5173 and common services on 3000/8000/8080)
- **Bind address**: `127.0.0.1` by default (localhost only; `--host` flag to override)
- **CORS**: `CORSMiddleware` with `allow_origins=["*"]` — acceptable since the server binds to localhost by default. Revisit if/when remote hosting is added.
- **Concurrency**: The generate endpoint is defined as a regular (non-async) function so FastAPI automatically runs it in a threadpool, keeping the event loop free for serving cached files and health checks. A threading lock serializes pipeline execution to avoid MediaPipe thread-safety issues and bound peak memory to one active run.
- **Lazy imports**: The `serve` subcommand handler imports `fastapi` and `uvicorn` inside the handler function, not at module level, so the CLI works normally for users who haven't installed the `[api]` optional dependency.

### Endpoints

**`GET /api/health`** — Health check

Returns `{"status": "ok"}`. Clients can use this to verify the server is running before attempting generation.

**`GET /api/maps/keys`** — Enumerate available map keys

Returns the list of requestable map keys with their descriptions and value ranges, sourced from `_MAP_DEFINITIONS`:
```json
{
  "maps": [
    {
      "key": "density_target",
      "value_range": [0.0, 1.0],
      "description": "How dark each region should be — density target for particle placement"
    },
    ...
  ]
}
```

**`POST /api/generate`** — Primary generation endpoint

Request body (JSON or multipart):
- `image`: file upload (multipart) OR `image_path`: string (local filesystem path)
- `maps`: list of requested map keys (e.g., `["density_target", "flow_x", "flow_y"]`). Must be from the 7 keys in `_MAP_DEFINITIONS`. If omitted, generate all available maps.
- `config`: optional object with per-pipeline parameter overrides, structured to mirror the actual dataclass hierarchy:
  - `features`:
    - `weights`: `{ "eyes": float, "mouth": float }` — maps to `PipelineConfig.weights`
    - `remap`: `{ curve, sigma, radius, tau, clamp_distance }` — maps to `PipelineConfig.remap` / `RemapConfig`
  - `contour`:
    - `contour_method`: string — maps to `ContourConfig.contour_method`
    - `direction`: string — maps to `ContourConfig.direction`
    - `band_width`: float | null — maps to `ContourConfig.band_width`
    - `smooth_contour`: bool — maps to `ContourConfig.smooth_contour`
    - `contour_thickness`: int — maps to `ContourConfig.contour_thickness`
    - `epsilon_factor`: float — maps to `ContourConfig.epsilon_factor`
    - `remap`: `{ curve, sigma, radius, tau, clamp_distance }` — maps to `ContourConfig.remap`
  - `density`:
    - `feature_weight`: float — maps to `ComposeConfig.feature_weight`
    - `contour_weight`: float — maps to `ComposeConfig.contour_weight`
    - `tonal_blend_mode`: string — maps to `ComposeConfig.tonal_blend_mode`
    - `tonal_weight`: float — maps to `ComposeConfig.tonal_weight`
    - `importance_weight`: float — maps to `ComposeConfig.importance_weight`
    - `gamma`: float — maps to `ComposeConfig.gamma`
    - `luminance`: `{ clip_limit, tile_size }` — maps to `ComposeConfig.luminance` / `LuminanceConfig`
  - `complexity`:
    - `metric`: string — maps to `ComplexityConfig.metric`
    - `sigma`: float — maps to `ComplexityConfig.sigma`
    - `scales`: float[] — maps to `ComplexityConfig.scales`
    - `scale_weights`: float[] — maps to `ComplexityConfig.scale_weights`
    - `normalize_percentile`: float — maps to `ComplexityConfig.normalize_percentile`
  - `flow`:
    - `coherence_power`: float — maps to `FlowConfig.coherence_power`
    - `fallback_threshold`: float — maps to `FlowConfig.fallback_threshold`
    - `contour_smooth_sigma`: float — maps to `FlowConfig.contour_smooth_sigma`
    - `blend_mode`: string — maps to `FlowConfig.blend_mode`
    - `etf`: `{ blur_sigma, structure_sigma, refine_sigma, refine_iterations, sobel_ksize }` — maps to `FlowConfig.etf` / `ETFConfig`
  - `flow_speed`:
    - `speed_min`: float — maps to `FlowSpeedConfig.speed_min`
    - `speed_max`: float — maps to `FlowSpeedConfig.speed_max`
- `persist`: optional string — if provided, also saves the bundle to disk at `output/{persist}/export/`. Value is sanitized to a safe directory name (alphanumeric, hyphens, underscores only).

Response (JSON):
```json
{
  "session_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "manifest": {
    "version": 1,
    "source_image": "portrait.jpg",
    "width": 1024,
    "height": 1365,
    "created_at": "2026-03-20T14:35:22.123456+00:00",
    "maps": [
      {
        "filename": "density_target.bin",
        "key": "density_target",
        "dtype": "float32",
        "shape": [1365, 1024],
        "value_range": [0.0, 1.0],
        "description": "How dark each region should be — density target for particle placement"
      },
      {
        "filename": "flow_x.bin",
        "key": "flow_x",
        "dtype": "float32",
        "shape": [1365, 1024],
        "value_range": [-1.0, 1.0],
        "description": "Flow field X component (unit vectors)"
      }
    ]
  },
  "base_url": "/api/maps/a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

The `manifest` field is the exact structure produced by `_manifest_to_dict()`, compatible with the plotter's `parseManifest()`. The `base_url` is the prefix for fetching individual maps: `{base_url}/{filename}` (e.g., `/api/maps/{session_id}/density_target.bin`). This matches the plotter's `MapBundle.load(baseUrl)` pattern where it fetches `{baseUrl}/manifest.json` and `{baseUrl}/{mapInfo.filename}`.

**`GET /api/maps/{session_id}/manifest.json`** — Fetch the manifest for a session

Returns the full `ExportManifest` JSON, identical to the `manifest` field in the generate response. This endpoint exists so that `MapBundle.load(baseUrl)` can fetch it by convention.

**`GET /api/maps/{session_id}/{filename}`** — Fetch a generated binary map

Returns raw `float32` binary data. Content-Type: `application/octet-stream`. The `filename` must match one of the `filename` values in the session's manifest (e.g., `density_target.bin`). Uses `FileResponse` to serve directly from the cache directory — no in-memory buffering. Serialization uses `array.astype(np.float32).tobytes()` via the same code path as `build_export_bundle` to guarantee byte-identical output.

**`GET /api/sessions`** — List active sessions

Returns active session IDs with metadata (source image, timestamp, requested maps). Useful for debugging and for clients that need to re-fetch.

**`DELETE /api/maps/{session_id}`** — Clean up a session's cached maps

Returns `204 No Content` on success. Returns `404` if the session doesn't exist.

### Image Handling

When `image_path` is provided, the server reads from the local filesystem. When `image` is uploaded as multipart, the server stores it in a temp directory for the duration of processing. This dual approach means:
- **Localhost development**: pass a path, no file copying needed
- **Future remote hosting**: upload the image, same API contract

The server validates that the image is loadable (OpenCV can decode it) and that MediaPipe detects at least one face before starting pipeline execution. If validation fails, the endpoint returns a `422` error with a descriptive message (e.g., "No face detected in image") rather than a `500` from a downstream pipeline crash.

### Disk-Backed Session Cache

Generated maps are written to a cache directory (`.cache/api/{session_id}/`) immediately after pipeline execution, then released from memory. The lifecycle:

1. Pipeline runs, produces numpy arrays in memory (unavoidable during computation)
2. Arrays are serialized to `.bin` files (float32) in the session's cache directory, along with `manifest.json`
3. Numpy arrays are released — memory is only held during the seconds of active pipeline execution
4. `.bin` and manifest endpoints serve files directly from the cache directory (`FileResponse`, no serialization overhead)
5. A background task evicts session directories after a configurable TTL (default: 30 minutes)

On server startup, the cache directory is scanned: any existing session directories are re-registered with fresh TTLs. This means a server restart doesn't orphan sessions — cached maps remain servable. Session IDs are UUID4, so collisions are not a concern.

The `persist` parameter copies the session's cache directory contents to `output/{name}/export/` and exempts the session from TTL cleanup. The value is sanitized to prevent path traversal.

### Pipeline Execution

The server determines which pipelines to run based on the requested maps. It calls the individual pipeline functions directly (not `run_all_pipelines`) and shares a single `LandmarkResult` across all pipelines that need it.

Map-to-pipeline dependencies:

| Requested map | Required pipelines |
|---|---|
| `density_target` | features + contour + density |
| `importance` | features + contour + density |
| `flow_x` | contour + flow |
| `flow_y` | contour + flow |
| `coherence` | contour + flow |
| `complexity` | complexity (standalone) |
| `flow_speed` | complexity + contour + flow |

The resolver unions the pipeline sets for all requested maps and runs each pipeline at most once.

### Dependency

FastAPI and uvicorn added as optional dependencies in `pyproject.toml` under `[project.optional-dependencies]`:

```toml
[project.optional-dependencies]
api = ["fastapi>=0.115", "uvicorn>=0.34"]
```

Installable via `uv pip install -e ".[api]"`. The core library remains dependency-free of web frameworks. If a user runs `serve` without the extras installed, they get a clear error message telling them to install `.[api]`.

## Risks and Considerations

- **Pipeline execution time**: Generating maps for a high-resolution image takes several seconds. The API is synchronous request/response, so the client blocks. For localhost single-user use this is acceptable. A future iteration could add async job-based generation, but that's premature now.
- **Memory during execution**: While memory is released after serialization to disk, peak memory during a single pipeline run can be 200-400 MB for a typical portrait image (multiple intermediate arrays for distance fields, influence maps, masks, etc.). The threading lock ensures only one pipeline runs at a time, bounding peak memory.
- **Disk usage**: The cache directory accumulates map data (~55 MB serialized output per full session, ~40 MB for a typical subset). TTL-based eviction and the explicit delete endpoint keep this bounded. A configurable max-sessions limit is a reasonable safeguard.
- **Image path security**: The `image_path` parameter reads from the local filesystem. This is fine for localhost but would need to be disabled or sandboxed if the server is ever exposed beyond localhost. The server binds to `127.0.0.1` by default.
- **MediaPipe thread safety**: MediaPipe's FaceLandmarker may not be thread-safe across all versions. The threading lock around pipeline execution prevents concurrent access. If this proves too conservative, it can be relaxed to a per-detector lock later.

## Non-Goals / Future Iterations

- **Preset/recipe system**: Named configurations that encode full pipeline parameter sets. Will layer on top of this API as an additional endpoint (e.g., `POST /api/generate` with `"preset": "high-contrast"` instead of inline config).
- **Composable map recipes**: Declarative blend-stack DSL for arbitrary map composition. Will extend the config schema.
- **Plotter-side integration**: Modifying the plotter repo's `MapBundle.load()` to fetch from this API instead of static files. Documented here so it can be built against this API's contract.
- **Watch mode / live reload**: Server watches for config file changes and pushes updated maps to connected clients. Would require WebSocket support.
- **Batch generation**: Generate maps for multiple images or multiple parameter sets in a single request.
- **Remote hosting**: Authentication, HTTPS, cloud deployment. The architecture supports this (upload-based image submission, stateless-ish design) but no work is done toward it now.
- **New exportable map types**: Adding maps like `tonal_target`, `contour_influence`, etc. to `_MAP_DEFINITIONS` would make them requestable through the API, but that's a separate decision about expanding the export contract.

## Success Criteria

1. `uv run python scripts/run_pipeline.py serve` starts a FastAPI server on `127.0.0.1:8100`.
2. A `POST /api/generate` request with an image path and `maps: ["density_target"]` returns a manifest and working URLs to fetch the binary map within 10 seconds for a typical portrait image.
3. The manifest in the response passes the plotter repo's `parseManifest()` without modification.
4. The returned `.bin` data is byte-identical to what the existing `--export` CLI flag produces for the same parameters.
5. Requesting a subset of maps (e.g., just `flow_x` and `flow_y`) only runs the pipelines needed (contour + flow), not the full suite.
6. The `persist` parameter writes an export bundle to disk that the plotter repo can consume from `public/maps/` without modification.
7. Swagger docs at `/docs` fully describe all endpoints, parameters, and response schemas.
8. A `GET /api/health` returns successfully, and `GET /api/maps/keys` returns the 7 available map keys.
9. An integration guide documents how a client would: check health, discover available map keys, generate a bundle, fetch individual maps, and optionally persist results.
